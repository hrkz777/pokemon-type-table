# 現状の設計指針

## 目的

このサイトは、ポケモンのタイプ相性を素早く確認できることを最優先とする。
主目的は装飾ではなく、一覧性・判読性・操作性である。

## デザイン基準

現状の見た目は、『ポケットモンスター ソード・シールド』の UI を土台にしている。
特に、以下の方向を基準とする。

- 白い情報面をベースにする
- 濃いダークカラーを UI の軸色として使う
- シアンとマゼンタをアクセントとして使う
- 情報ブロックを明確に分離する
- 選択状態は暗色ピルや強いアウトラインで示す
- タイプ色だけはポケモン固有の鮮やかさを維持する

参考イメージ:

- https://www.pokemon.co.jp/ex/sword_shield/system/
- https://www.pokemon.co.jp/ex/sword_shield/howtoplay/
- https://www.pokemon.co.jp/ex/sword_shield/howtoplay/191206_01.html

## ビジュアル方針

### 1. 剣盾らしい整理された UI にする

- 背景は明るいグレー系の無地を基本とする
- 主コンテナは白い面で構成する
- ダークグレーまたは黒系の UI パーツを軸として使う
- アクセント色は全面に撒かず、絞って使う

### 2. アクセント色はシアンとマゼンタを基準にする

- シアンはフォーカス、選択、補助アクセントに使う
- マゼンタは見出し面や強い視線誘導に使う
- 黄色は補助色としてのみ使う
- UI 全体をカラフルにしすぎない

### 3. フラット寄りを維持する

- 面は単色を基本とする
- 強いグラデーションは使わない
- ガラス風、メタリック風、過剰な立体感は使わない
- 情報整理は色数ではなく、境界線とレイアウトで行う

### 4. タイプ色の可読性を最優先する

- タイプバッジは鮮やかな背景色を前提とする
- 文字アウトラインは必須とする
- フラット化やミニマル化を理由に、タイプバッジの文字アウトラインを消してはならない

## レイアウト方針

### 1. 上部構成

- 先頭には独立したヒーローブロックを置く
- ヒーローは白面を基調とし、剣盾らしいアクセント形状を添える
- タイトルはダークカラーのピル形状で表示する
- ヒーロー自体が主張しすぎないように、情報面とのバランスを取る

### 2. 全体コンテナ

- メインコンテンツは白い 1 枚のシェル内に収める
- 角丸は中程度に留める
- 影は極小にし、浮遊感を出しすぎない

### 3. タブ

- タブは横並びのピル型を基本とする
- 未選択は白面 + 薄い境界線
- hover はシアン寄りの変化
- 選択中はダークカラー面 + 白文字で示す
- 選択中だけを明確に強く見せる

### 4. 相性表

- 相性表はこのサイトの中心要素である
- 上部ヘッダーはダークカラー背景 + 白文字
- 上部ヘッダー下端にはシアンの強いアクセント線を入れる
- 左側の軸ラベルもダークカラー背景 + 白文字で統一する
- 表面は白を基本にし、交互背景はごく薄くする
- hover はシアン系の淡い補助色だけに留める
- sticky header / sticky side column は維持する

### 5. 自分のポケモン用パネル

- タイプ選択と結果表示は、表と同じ世界観の白面 UI でまとめる
- 表より目立たせず、補助 UI として整理する
- ただし選択状態だけははっきり見せる

## コンポーネント方針

### タイプバッジ

- タイプバッジはこのサイトの視覚的アイコンである
- 色は `types.css` のタイプ別変数を使用する
- 文字アウトラインは維持する
- バッジの鮮やかさは下げない
- UI 全体が落ち着いている分、タイプ色は情報の核として残す

### タイプ選択ボタン

- バッジそのものがボタンとして振る舞う構造を維持する
- 余計な外枠や別レイヤーを足さない
- 状態は以下で表現する
	- 未選択: 不透明
	- 選択中: 不透明 + 強いシアンアウトライン
	- 2つ選択済み時の未選択: 半透明

### 見出し

- セクション見出しはダークカラーのピル型を基本とする
- シアンの小アクセントを添える
- 見出しだけで剣盾らしさが分かる状態を目指す

### 相性カード

- 4倍、2倍、等倍、0.5倍、0.25倍、0倍ごとに分類する
- カード本体は白寄りまたは淡色面とする
- 色差はカード上端のアクセントラインと薄い背景色で示す
- 強い立体表現は避ける
- 色分けは補助情報であり、文字情報を読めなくしてはならない

## 余白・角丸・影

- 余白は広すぎず詰めすぎず、ゲーム UI の情報密度に寄せる
- 角丸は中程度に留める
- 影は最小限にする
- テーブル周辺では特に平面的に見せる

## タイポグラフィ方針

- ベースフォントは `Noto Sans JP`
- 見出しは太め、本文は標準
- 文字サイズ差は必要最小限に留める
- 見た目の演出よりも判読性を優先する
- タイプ名は色面に負けない可読性を確保する

## 実装上の境界

### CSS

- 全体レイアウトと UI の見た目は `docs/css/styles.css`
- タイプ固有色は `docs/css/types.css`
- タイプごとの配色は `types.css` に集約し、個別インライン style は使わない

### JavaScript

- 画面構築は `docs/js/app.js` が行う
- `index.html` は最小シェルとし、UI の実体は JavaScript で組み立てる
- CSS を前提に DOM を作るため、クラス名の意味を崩さない

### データ

- 非テキストデータは `docs/assets/`
- ローカライズ文言は `docs/locales/<locale>/`
- UI 文言は JavaScript に直接埋め込まない

## 現時点で避けるべきこと

- 意味の薄いグラデーション
- ガラス風 UI
- 過剰な影
- 情報面を無意味に色分けしすぎること
- UI 全体をタイプ色で塗りつぶすこと
- タイプ色の弱体化
- タイプバッジの文字アウトライン削除
- 抽象的な Material 風装飾へ戻すこと
- 公式タイプ色より UI 装飾を優先すること

## 次の再設計で判断基準にすること

- 相性表が最優先で読みやすいか
- 剣盾らしいダーク UI とアクセント色の整理ができているか
- タイプバッジの視認性が落ちていないか
- 自分のポケモン用 UI が主役を邪魔していないか
- 装飾ではなく情報整理として正しい変更か
